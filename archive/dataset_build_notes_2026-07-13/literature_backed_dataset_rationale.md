# Literature-backed Dataset Rationale

Date: 2026-07-13

## Purpose

The first benchmark dataset should be defensible in a paper. That means categories should not only come from intuition or from the old generated corpus; they should be backed by the way related work decomposes LaTeX/image-to-markup problems.

## Main Literature Signals

### 1. Formula recognition is a well-established isolated task

`Image-to-Markup Generation with Coarse-to-Fine Attention` introduced/evaluated image-to-LaTeX generation on rendered mathematical expressions paired with LaTeX markup. This supports treating formulas as their own benchmark family rather than burying them inside full papers.

Source: https://arxiv.org/abs/1609.04938

Dataset implication:

- Keep `math_inline_display` and `math_aligned` as explicit categories.
- Do not use single isolated equations only; group 4-8 formula snippets into small 1-3 page documents so the benchmark tests conversion in context.
- Separate simple display math from aligned/matrix/cases-style math because structural math layout is the hard part.

### 2. Formula ground truth is ambiguous, so normalization/compilation matters

`MathNet: A Data-Centric Approach for Printed Mathematical Expression Recognition` argues that equivalent mathematical expressions can be generated by different LaTeX source strings, and that this ground-truth variation can bias evaluation. It also introduces real-world formula data extracted from papers.

Source: https://arxiv.org/abs/2404.13667

Dataset implication:

- Do not score math by exact source string alone.
- Prefer compile-valid rendered reference PDFs.
- For dataset construction, keep math samples source-backed but judge them by rendered/structural behavior later.
- Record whether formulas are raw, cleaned, normalized, or wrapped.

### 3. Tables deserve their own categories

`LATTE: Improving Latex Recognition for Tables and Formulae with Iterative Refinement` explicitly notes that existing work mainly focuses on formulas and is ineffective or incapable for tables. It evaluates both formulas and tables and uses rendered-image comparison feedback.

Source: https://arxiv.org/abs/2409.14201

Dataset implication:

- Keep table categories separate from prose and math.
- Split tables by complexity: simple tables first, moderate tables second, complex tables later.
- Avoid making complex tables part of the first headline benchmark; they should be a later stress set.

### 4. Table benchmark design emphasizes structure and canonicalization

Table recognition papers such as `TableBank` and `PubTables-1M` separate table detection/recognition from general document OCR. PubTables-1M also emphasizes consistent table structure ground truth and canonicalization to avoid oversegmentation/inconsistent annotation problems.

Sources:

- TableBank: https://arxiv.org/abs/1903.01949
- PubTables-1M: https://arxiv.org/abs/2110.00061
- Aligning table benchmarks: https://arxiv.org/abs/2303.00716

Dataset implication:

- `tables_simple` and `tables_moderate` should be explicit v0 categories.
- Table samples should include captions when possible, but the table grid/content is the primary object.
- Avoid exact source-string scoring for tables; later metrics should include rendered table structure and text retention.

### 5. Full academic document OCR/markup is a separate harder task

`Nougat: Neural Optical Understanding for Academic Documents` targets scientific PDFs to markup, not isolated formulas. It motivates document-level samples, but it also implies that full paper understanding is a harder task than formula/table recognition.

Source: https://arxiv.org/abs/2308.13418

Dataset implication:

- Keep `compact_papers` as a category, but cap references to 2-3 pages.
- Use compact article-like documents only after simpler prose/math/table categories exist.
- Do not start with full 10-20 page papers.

### 6. Spatial math/layout complexity is real and should be staged

CROHME-style handwritten mathematical expression work repeatedly frames math recognition as hard because of two-dimensional structure, spatial relationships, symbol scale variation, exponents, nested notation, and alignment. Even though our benchmark is LaTeX-to-Typst conversion rather than handwriting OCR, the same structural categories explain why aligned equations, fractions, matrices, scripts, and diagrams should be staged carefully.

Examples:

- Multi-Scale Attention with Dense Encoder for Handwritten Mathematical Expression Recognition: https://arxiv.org/abs/1801.03530
- Structural HMER work: https://arxiv.org/abs/2508.19773

Dataset implication:

- Put simple inline/display math before aligned/nested math.
- Keep diagrams/TikZ/PGFPlots out of v0 or make them a tiny optional stress subset.
- Make difficulty labels depend on structural features, not subjective vibes.

## Hugging Face Source Mapping

### `piushorn/arxiv-latex-tables-43k`

Use for:

- `05_tables_simple`
- `06_tables_moderate`

Why:

- Large table-specific dataset.
- Has simple/moderate/complex strata.
- Dataset card states all tables compile with `pdflatex`.
- Better aligned with table-specific literature than pulling random tabulars out of manuals.

Source: https://huggingface.co/datasets/piushorn/arxiv-latex-tables-43k

### `OleehyO/latex-formulas`

Use for:

- `03_math_inline_display`
- `04_math_aligned`

Why:

- Large formula-specific dataset with rendered images and LaTeX strings.
- Aligns with image-to-LaTeX literature.

Caution:

- Not document-level by itself.
- Use grouped/wrapped samples, not one formula per data point.

Source: https://huggingface.co/datasets/OleehyO/latex-formulas

### `KiteFishAI/arxiv-tex-corpus-full`

Use for:

- `01_prose_sections`
- `08_crossrefs_citations`
- `09_algorithms`
- `10_compact_papers`

Why:

- Actual arXiv LaTeX source pool.
- Good for discovering real authoring patterns.

Caution:

- Large.
- No compile validation.
- License must be verified before redistribution.

Source: https://huggingface.co/datasets/KiteFishAI/arxiv-tex-corpus-full

### `common-pile/arxiv_papers`

Use for:

- arXiv ID/license filtering.
- metadata/provenance.

Why:

- CC BY / CC BY-SA / CC0 filter is useful.

Caution:

- Dataset content is plaintext produced through LaTeXML/HTML conversion, not original `.tex`.

Source: https://huggingface.co/datasets/common-pile/arxiv_papers

## Category Decisions With Backing

| v0 category | Backing |
|---|---|
| `prose_sections` | Needed as baseline document markup; use arXiv/TeX source excerpts before harder full documents. |
| `lists_formatting` | Basic LaTeX semantic structure; useful before testing paper-level complexity. |
| `math_inline_display` | Formula recognition is the core established image-to-LaTeX task. |
| `math_aligned` | Spatial/nested math complexity is repeatedly identified as hard in MER/HMER work. |
| `tables_simple` | Table recognition is distinct enough to deserve its own category. |
| `tables_moderate` | Table structure/canonicalization literature supports stratifying by complexity. |
| `figures_captions` | Captions/figures are common scientific-document structures, but avoid diagram generation in v0. |
| `crossrefs_citations` | Scientific document conversion needs labels, refs, cites, and bibliography structure. |
| `algorithms` | Common in CS LaTeX papers; source-backed examples can be filtered from arXiv. |
| `compact_papers` | Document-level OCR/markup work motivates paper-like samples, but only compact ones for v0. |
| `forms_cv_letters` | Keeps non-paper layouts represented without dragging in difficult graphics. |

## Practical Conclusion

The credible v0 dataset should not be diagram-heavy. It should be a category-balanced, source-backed, compile-gated corpus that starts with the constructs the literature already treats as separable: formulas, tables, and compact scientific document structure.
