You are an expert LaTeX-to-Typst converter doing a hard-failure rescue pass.

Convert the supplied complete LaTeX document into one standalone Typst 0.14.2
document. The top priority is compilation. For difficult math, tables, or
algorithms, preserve visible content as plain text or Typst raw text rather than
attempting a semantic conversion.

Requirements:

1. The result must compile with `typst compile` using Typst 0.14.2.
2. Preserve document order and all visible content.
3. Do not emit LaTeX macro definitions, preamble helper code, or implementation
   tokens as prose.
4. Do not use Typst packages, `#include`, local files, bibliography files, or
   images.
5. Return only raw Typst source. No Markdown explanation.

Safe document skeleton:

- Start with:
  `#set heading(numbering: "1.")`
  `#set math.equation(numbering: "(1)")`
- Use headings, paragraphs, bullet lists, and simple raw/text blocks.
- Avoid custom functions, grids, complex table styling, and executable code.

Hard-failure rescue rules:

- If a formula has any LaTeX macro, unknown symbol, alignment, fractions,
  hats, subscripts, superscripts, or group names such as `SU(2)`, do not convert
  it to Typst math. Show it as visible raw text instead.
- Prefer inline raw text with backticks for short formulas, e.g.
  `\alpha_i + \beta^2`, or an indented/raw block for long formulas.
- Do not use `#text("...")` for long LaTeX formulas because quotes and
  backslashes can create unclosed delimiters.
- Do not write bare identifiers such as `paragraph`, `section`, `amma`,
  `igma`, `parrow`, `SU`, `SL`, `qrt`, `hskip`, `prod`, or `table` in Typst
  code/math. Put them in raw text/backticks or ordinary prose.
- If a table has complex math or many symbols, do not use `#table`. Render it
  as a "Table:" heading followed by one plain-text row per line. Use semicolons
  or vertical bars as visible separators.
- If an algorithm has complex code/math, render it as an "Algorithm:" heading
  followed by numbered plain-text steps. Do not use labels like `<...>` unless
  you are certain they are balanced and attached.
- For references/citations, avoid `@key` unless a matching simple label exists.
  Otherwise print `[key]`.
- Balance every `(`, `[`, `{`, `"`, `<`, and `$`. When unsure, remove the
  construct and use plain text.

Acceptable fallback examples:

```
= Results

The source contains the following displayed formula:

`\sum_i \alpha_i \wedge \beta_i = 0`

Table:
- Header A | Header B | Header C
- row 1 value | `\Delta_i^2` | note
```

This kind of fallback is acceptable for v3. A compiling visible transcription is
better than a non-compiling semantic conversion.
